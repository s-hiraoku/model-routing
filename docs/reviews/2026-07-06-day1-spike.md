## 指摘事項

### 1. [重要] レスポンスの content-encoding 不整合が偽陰性を招く恐れ

`scripts/spike-rewrite.ts:128-134` — 上流の `Response` をそのまま返している。Bun の `fetch` はレスポンスボディを自動解凍する一方、返却 Response に上流の `content-encoding: gzip` / `content-length`(圧縮時の値)ヘッダが残る可能性がある。その場合クライアント(Claude Code)は解凍済みボディを再解凍しようとして失敗する。

**これが致命的なのは、失敗の見た目が「書き換えたら壊れた」になること。** プロキシ実装のバグを「OAuth では model 書き換え不可」という誤った No-Go 判定として記録してしまうと、プロジェクト全体の進路を誤る。

対応: 上流レスポンスを包み直し、`content-encoding` / `content-length` / `transfer-encoding` をヘッダから除去する。

```ts
const upstreamRes = await fetch(...);
const headers = new Headers(upstreamRes.headers);
headers.delete("content-encoding");
headers.delete("content-length");
headers.delete("transfer-encoding");
return new Response(upstreamRes.body, {
  status: upstreamRes.status,
  statusText: upstreamRes.statusText,
  headers,
});
```

### 2. [重要] チェック項目「レスポンスの model が書き換え後」を確認する手段がない

docs/03 のスパイクチェックリストは「200 が返り、**レスポンスの `model` が書き換え後になっていること**」を要求しているが、`claude -p` の出力にモデル名は出ず、プロキシもレスポンス側を一切ログしない。現状ではチェックリストを完遂できない。

対応: プロキシに数行追加し、`/v1/messages` のレスポンスから model をログする。

- 非ストリーミング: レスポンス JSON の `model`
- ストリーミング: 最初の `message_start` イベント内の `message.model` をスニフ(tee して先頭チャンクだけ覗く。中継はブロックしない)

### 3. [中] 対照実験(パススルー)がチェックリストにない

docs/03 の手順は「passthrough で通常動作確認**後**、書き換え実験」。`docs/decisions.md` のチェックリストに対照ステップがないため、書き換え実験が失敗したとき「プロキシ起因」か「書き換え起因」かを切り分けられない。

対応: decisions.md のチェック先頭に追加 —
`[ ] REWRITE_MODEL 未設定(パススルー)で claude -p が streaming 含め正常動作(対照)`

### 4. [軽] hop-by-hop ヘッダ未除去

`connection` / `keep-alive` / `te` 等の hop-by-hop ヘッダを上流へ転送している。Bun の fetch が無視する可能性が高く、スパイクでは実害は薄い。M0 本実装では除去を明示すること(docs/03 のログ許可リストとは別に、転送ヘッダの扱いとして)。

### 5. [軽] `@types/bun` が devDependencies にない

`tsconfig.base.json` が `"types": ["bun-types"]` を指定しているが依存が未導入(lockfile もなし)。`bun test` は動くが、エディタ・`tsc` の型チェックが機能しない。`bun add -d @types/bun`(現行の推奨パッケージ名は導入時に確認)+ lockfile をコミット。

### 6. [軽] 非 `/v1/messages` の POST も全ボディをバッファ

`scripts/spike-rewrite.ts:110` — 書き換え対象外のリクエストまで `req.text()` で全読みしている。スパイクでは許容。M0 ではストリーム素通しにすること(docs/03 のフロー準拠)。

## 良かった点
