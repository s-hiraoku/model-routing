## 1. スパイク結果の設計インパクト(最重要)

`docs/decisions.md` に記録された結果:

| 実験 | 結果 |
|---|---|
| パススルー対照 | ✅ 正常(streaming 込み) |
| 昇格(fable-5 → opus-4-8) | ✅ 200。レスポンス model も書き換え後 |
| 降格(fable-5 → haiku-4.5) | ❌ **400: `This model does not support the effort parameter`** |

**設計が想定した縮退パターンの逆が起きた。** docs/03 は「昇格だけ拒否される場合は降格専用機に縮小」と書いたが、現実は昇格成立・降格不成立。原因は OAuth ではなく**モデル間のパラメータ互換性**(Claude Code が送る `effort` を Haiku が受けない)。つまり「model のみ書き換え」という現行設計の降格が成立しない。

### 提案(設計者として)

原因が認証ではなくパラメータ互換なので、対処可能な見込みが高い:

1. **スパイク追試**: 降格時に `model` 書き換え + **非互換パラメータ(`effort`)を除去**したリクエストが 200 になるか検証する(除去対象はターゲットモデルが受けないパラメータなので、挙動を変えるのではなく「ターゲットの仕様に合わせる」正当な変換)
2. 通れば docs を更新: models.yaml の tier 定義に `strip_params: [effort]` のような**パラメータ互換レイヤ**を追加し、docs/03(書き換え処理)と docs/05(demote パス)に反映
3. 通らなければ「昇格専用機」として再設計(枠温存の柱が弱まるため、プロジェクトの費用対効果を再評価)

**この追試が終わるまで M4 の降格ロールアウト前提は凍結**(decisions.md の Route decision と整合)。

## 3. 実装済みコードへの指摘

### 3-1. [中] gateway に上流エラーハンドリングがない

`packages/gateway/src/app.ts:24-34` — `proxyToUpstream` の fetch が reject した場合(上流ダウン・DNS 失敗等)、ハンドリングがなく Hono の汎用 500 になる。スパイク版にはあった try/catch が gateway 版で消えている。502 + JSON を返し、将来 `status='gateway_error'` として記録できる形に。

### 3-2. [中] スキーマ定義の二重管理

`packages/datastore/src/schema.ts`(Drizzle)と `src/init.ts`(生 SQL)が同じスキーマを二重に定義している。docs/01 の設計は「Drizzle スキーマ + drizzle-kit でマイグレーション管理」。今後テーブルが増えるたびに 2 箇所の同期が必要になり、確実にズレる。`drizzle-kit generate` で SQL を生成する構成に寄せるか、当面 Drizzle スキーマを正として init は生成物を読む形にすること。

### 3-3. [軽] hop-by-hop ヘッダ未除去(前回 #4 の持ち越し)

スパイクでは「M0 本実装で対応」とした項目。gateway が本実装なので、`buildUpstreamHeaders` で `connection` / `keep-alive` / `te` / `upgrade` / `proxy-*` を除去すること。

### 3-4. [軽] mode 判定の評価タイミングが不統一

`main.ts:22-23` は起動時に env を固定、`app.ts:42` は healthz でリクエスト毎に `process.env` を参照。現状実害はないが、キルスイッチの意味論(起動時か実行時か)を決めて統一すること。設計意図は「実行時に効く」なので、判定関数を shared に置いてリクエスト毎評価に寄せるのが正。

### 3-5. [軽] datastore / db-init が未コミット

動作しているならコミットすること(レビューは働きかけたが、未コミットの作業ツリーはレビュー対象として不安定)。
