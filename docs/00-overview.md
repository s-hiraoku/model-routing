# 00. プロジェクト概要とロードマップ

## コンセプト: Claude Code のオートマチックトランスミッション

**ユーザーがモデルを一切意識しなくても、タスクに応じて最適な Claude モデルが選択されてアウトプットが出る体験**を作る。

車の AT と同じ発想。ギア(Opus / Sonnet / Haiku)を手で選ばず、負荷に応じて自動変速する:

- 設計・難しいデバッグ → **Opus に自動シフトアップ**(品質が要る場面で勝手に上がっている)
- 通常の実装 → **Sonnet**(デフォルト)
- ツール実行の中間ターン・軽い雑務 → **Haiku に自動シフトダウン**(枠を温存)

ユーザー体験上の変化は「Opus 品質が必要な場面で勝手に Opus になっている」「Max サブスクの枠が減りにくくなった」だけ。それ以外は今まで通り Claude Code を使うだけ。

### 本プロジェクトのキモ: 自己進化

モデルは次々と世代交代する。だから変速ロジックは静的なルール集ではなく、**ログを常に参照して自分で最適化し続ける自己進化システム**として設計する:

- **自動ループ**: 夜間のログ監視(暗黙シグナル・新モデル検知・劣化検知)+ 週次の評価バッチ → ポリシー自動再生成 → 劣化時は自動ロールバック
- **人間ループ**: 人間の意思を「注意予算」内(A/B 選好質問は週 3 問・1 問 1 分、満足度確認は月 1 問)で吸い上げ、自由記述の要望は LLM がポリシー変更案に翻訳して人間が承認。overrides は絶対的拒否権として何世代進化しても保持

詳細は [07-feedback-and-evolution.md](07-feedback-and-evolution.md)。

### スコープ決定の経緯(重要)

当初はマルチプロバイダー(Claude/GPT/Gemini)構想だったが、以下の理由で **Claude ファミリー内に限定**した:

1. **サブスクのみ(従量課金 API 不使用)の制約**下では、他社モデルは各社 CLI(Codex/Gemini CLI)経由でしか呼べず、リクエスト経路の内側に入れられない
2. タスク単位で他社 CLI へ委譲する方式は「継ぎ目」が必ず見える = **「モデルを意識しない」という本来の目的と矛盾する**
3. Claude 内なら、Claude Code → api.anthropic.com のトラフィックをローカルプロキシで素通しし、`model` フィールドを書き換えるだけで**完全に透明な変速**が実現できる

## 成立条件(最初に検証する)

この構想には未検証の前提が 1 つある:

> **サブスク(OAuth)認証のリクエストで `model` フィールドを書き換えても API に受理されるか**

これが通らなければプロジェクト全体が成立しない。よって**実装初日に curl レベルのスパイクで検証する**(M0 参照)。通らなかった場合の Plan B は「ログ・可視化ツールとして完成させる」に縮退。

## 基本方針

1. **評価が先、変速が後。** 「どのターンを Haiku に降ろしても壊れないか」「どのタスクは Opus に上げる価値があるか」を自分の実開発ログで検証してから変速ルールを有効化する。
2. **実トラフィックからサンプリングして評価する。** 実際に投げたタスクを worktree で再実行し、モデル階級(tier)間で成果を比較する。
3. **評価リプレイも本番の gateway を通す。** 変速ポリシーを適用した状態のリプレイができるので、「ルール単体」ではなく「本番と同一の挙動」を検証できる。
4. **確信がないときは触らない。** 変速はオプトイン動作。判定に迷うリクエストは全部素通し。
5. **壊れても開発が止まらない。** キルスイッチ・パススルーモード・環境変数を外せば素の接続、を常に維持。

## 全体アーキテクチャ

```
┌─────────────┐  ANTHROPIC_BASE_URL
│ Claude Code  │ ─────────────────────┐
│ (普段の開発)  │                      ▼
└──────┬──────┘             ┌──────────────────┐
       │ hooks               │  Gateway (proxy)  │        api.anthropic.com
       │ (task境界・git HEAD) │  ─ logger         │ ──────▶ (OAuth 素通し・
       ▼                     │  ─ shifter(変速機) │          model のみ書き換え)
┌──────────────┐            └────────┬─────────┘
│ /internal/    │◀────────────────────┤
│  task-event   │                     ▼
└──────────────┘             ┌────────────┐        ┌──────────────────────┐
                             │  SQLite DB  │◀──────▶│  Eval Pipeline        │
                             │  (全ログ)   │        │  sample → replay      │
                             └────────────┘        │  (Agent SDK×gateway経由)│
                                                   │  → judge → aggregate   │
                                                   └──────────┬───────────┘
                                                              ▼
                                                    ┌────────────────┐
                                                    │  Review UI      │ 人間の抜き取り検証
                                                    └────────────────┘
```

## フェーズとマイルストーン

| フェーズ | 内容 | Go/No-Go ゲート |
|---|---|---|
| **M0** | **OAuth model 書き換えスパイク(初日)** → 透過プロキシ + 全ログ + hooks | 書き換えが通る(通らなければ Plan B へ)。1 週間プロキシ経由で支障なし |
| **M1** | ログ探索 + タスク分類器 + サンプラー | 分類の目視納得率 80% 以上。サンプル母数が足りる |
| **M2** | tier 別リプレイ + LLM-as-Judge + Review UI | 人間とジャッジの一致率 κ ≥ 0.6 |
| **M3** | 集計 + レポート + 変速ポリシー生成 | 「Haiku 降格しても安全なカテゴリ」「Opus 昇格の価値があるカテゴリ」が最低 1 つずつ判定できる(なしという結論も可) |
| **M4** | 変速の本番有効化(降格 → 昇格の順に段階投入)+ changelog 通知 | 1 週間で体感品質の劣化なし・エラー率 < 1%・キャッシュ損 < 変速益 |
| **M5** | **自己進化ループ**(夜間監視・週次自動再生成・A/B 選好プッシュ・`/model-feedback`・自動ロールバック・新モデル検知) | A/B 質問の回答率 ≥ 60%。フィードバック→反映 ≤ 1 週間。モデル世代交代を設定 1 行 + 自動再評価で乗り切れる |

## 想定リスクと対策(「やってみたら違った」候補)

| リスク | 起きること | 対策 |
|---|---|---|
| **OAuth で model 書き換えが通らない** | 構想全体が不成立 | 初日スパイクで最速判定。Plan B(ログ・可視化ツール)へ縮退 |
| **プロンプトキャッシュの分断** | ターン単位でモデルを切り替えるとキャッシュが効かず、かえって枠消費が増える | 変速はタスク境界優先・ターン内は連続区間でまとめて降格。M4 で cache_read_tokens を before/after 計測し、損 > 益なら降格ルールを縮小 |
| **Claude Code 本体の機能と被る** | `/model opusplan` や雑務の自動 Haiku 化など、本体が既にやっている部分と重複 | 本体がやらない領域に集中: ターン単位降格、タスク内容での自動昇格、枠残量連動。被った機能は素直に本体に譲る |
| **ジャッジが人間と一致しない** | 評価が信用できない | M2 の κ ゲート。Claude 同士の比較なので自己選好バイアスは相対的に小さいが、位置バイアス対策(A/B 入替)と人間抜き取りは維持 |
| **Haiku 降格の失敗が見えにくい** | ツール呼び出しの質が微妙に下がり、ターン数が増えて逆に枠を消費 | 評価指標に「タスク完了までの総ターン数・総トークン」を含める。本番でも変速あり/なしでターン数を比較監視 |
| **評価が開発枠を食う** | リプレイ・ジャッジも同じ Max 枠 | 夜間・週末のみ実行、バッチ上限、429 検出で中断・次ウィンドウ再開 |
| **分類ノイズ** | 誤昇格・誤降格 | 迷ったら素通し(現状維持)に倒す。昇格・降格の閾値を非対称に(降格は保守的に) |
| **フィードバック疲れ** | 質問がうるさくなった瞬間、人間ループ全体が無視されて死ぬ | 注意予算制(週 3 問・月 1 問・期限切れは督促なし)。未回答は「データなし」扱いで同意とみなさない。回答率 60% を切ったら頻度を下げる |
| **自己進化の暴走** | 自動再生成が毎週ポリシーを振り回して挙動が安定しない | 変更予算(1 サイクル 2 変更まで)、全変更に根拠 + changelog、自動ロールバックの復帰は再検証必須、overrides は常に最優先 |
| **プロキシ不具合で開発停止** | 本末転倒 | パススルーモード、`MODEL_ROUTING_DISABLED=1`、環境変数を外せば素の接続 |

## 技術スタック(決定事項)

- **ランタイム/ツールチェーン**: **Bun**(runtime + workspaces + test runner + `bun:sqlite`)
- **言語**: TypeScript(ESM only)
- **HTTP**: Hono v4(Review UI は Hono JSX で SSR)
- **DB**: SQLite(`bun:sqlite`)+ Drizzle ORM
- **バリデーション**: Zod v4 / **Lint・Format**: Biome 2
- **評価実行**: Claude Agent SDK(`@anthropic-ai/claude-agent-sdk`。サブスク認証を継承、`ANTHROPIC_BASE_URL` で gateway 経由にできる)
- 外部 API SDK・他社 CLI は使わない
