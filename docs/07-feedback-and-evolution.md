# 07. 自己進化ループと人間フィードバック(本プロジェクトのキモ)

モデルは次々と世代交代し、自分のワークロードも変わる。だから変速ロジックは**一度作って終わりではなく、ログを常に参照して自分で最適化し続ける**。ただし最適化の主権は人間にあり、**人間の負荷にならない頻度**で意思を吸い上げて反映する。

## 二重ループ構造

```
┌────────────────────── 自動ループ(データ駆動)──────────────────────┐
│                                                                      │
│  毎夜: ログ監視(暗黙シグナル・ドリフト・新モデル検知)                  │
│  毎週: 評価バッチ(新サンプル + 新モデル)→ ポリシー再生成              │
│  随時: 本番劣化検知 → 自動ロールバック                                 │
│                                                                      │
└───────────────┬──────────────────────────────▲──────────────────────┘
                │ 提案・変更通知・質問(注意予算内)  │ 承認・選好・要望
┌───────────────▼──────────────────────────────┴──────────────────────┐
│                     人間ループ(意思反映)                              │
│                                                                      │
│  A/B 選好質問(プッシュ、週3問まで)/ 満足度チェック(月1問)            │
│  /model-feedback(自由記述、いつでも)/ overrides(絶対的拒否権)       │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

原則: **自動ループは「証拠を作って提案する」、人間ループは「方向を決めて拒否権を持つ」。** ポリシーの全変更は根拠付きで changelog に記録され、人間はいつでも巻き戻せる。

## 注意予算(attention budget)

人間フィードバックの最大の敵は**フィードバック疲れ**(うるさくなった瞬間に無視されて死ぬ)。よって人間への割り込みを予算制にする。

### config/feedback.yaml

```yaml
attention_budget:
  preference_questions_per_week: 3    # A/B 選好質問の上限
  question_expire_days: 7             # 答えなかった質問は静かに期限切れ(督促しない)
  satisfaction_check_interval_days: 30
  quiet_hours: [9, 10, 11, 14, 15, 16, 17]   # 集中時間帯は通知しない

notify:
  method: macos                       # osascript の通知(クリックで review-ui が開く)

changes:
  max_rule_changes_per_batch: 2       # 1 サイクルでのポリシー変更数上限(暴走防止)
  keep_policy_versions: 20            # ロールバック用に保持する世代数

implicit_signals:                     # 暗黙シグナルの閾値(自動ロールバック判定)
  correction_rate_jump: 1.5           # 変速有効カテゴリの訂正率が基準の 1.5 倍超
  min_n: 20
```

**未回答は「無言の承認」ではなく「データなし」として扱う**(勝手に同意とみなさない)。答えたくなければ答えなくてよい、が成立していることが継続の条件。

## 人間ループ: 4 つのチャネル

### 1. A/B 選好質問(プッシュ型。チャットの「どちらの回答がいいですか?」に相当)

週次バッチのジャッジ済みペアから、**人間の一票が最も価値を持つペア**を選んで通知する:

```
優先度スコア = 影響度 × 不確実性
  影響度   : そのカテゴリの実トラフィック量 × ポリシー変更の当落線上にあるか
  不確実性 : position 入替で判定が割れた / win_rate が閾値ギリギリ / κ が低いカテゴリ
```

- 通知: 「モデル比較が 1 件あります(約 1 分)」→ クリックで review-ui のクイック比較ページ
- 画面はブラインド A/B(既存の Review UI 比較画面と同一)。回答は `human_reviews(source='push')` に記録
- **人間の一票はそのペアのジャッジ判定を上書き**する(集計時に judge 判定を破棄して human 判定を採用)。同時に κ の継続計測にも使う
- 週の予算を使い切ったら、それ以上のレビュー対象は従来どおり review-ui のキューに静かに積まれるだけ(見たい人だけ見る)

### 2. 自由記述フィードバック(プル型。いつでも)

```bash
# Claude Code 内から(skills/model-feedback/SKILL.md)
/model-feedback 最近 plan 系で Opus に上がってない気がする。もっと積極的に上げてほしい

# ターミナルから
bun run feedback -- add "docs を Haiku にするのは賛成だけどコミットメッセージは Sonnet がいい"
```

処理フロー(週次バッチの stage 7):

```
feedback_notes(pending)
  → LLM(Agent SDK)が解釈: 意図・対象カテゴリ・具体的なポリシー変更案に翻訳
     例: "コミットメッセージは Sonnet がいい"
       → 提案: docs カテゴリを細分化 or overrides に「docs のうち commit 文脈は mid 固定」
  → 変更案を review-ui の「提案」ページ + 通知で提示
  → 人間が承認 → ポリシーに反映(origin: human_feedback)/ 却下 → 記録のみ
```

**フィードバックが評価の設計自体を変えることもある**(カテゴリ分割、新しい評価観点)。その場合は「次バッチでこの観点のサンプルを厚くする」という形で評価計画に還流する。

### 3. 満足度チェック(月 1 問)

月に一度だけの一問通知: 「この 1 ヶ月の自動変速に不満はありましたか?」

- 選択肢: `満足` / `やや不満(自由記述へ)` / `変速を止めたい`
- `やや不満` → feedback_notes に入りチャネル 2 と同じ処理
- `止めたい` → 即座に該当機能を passthrough 化(問答無用。拒否権の実装)

### 4. overrides(絶対的拒否権)

`shift-policy.yaml` の `overrides` は自動再生成でも常にマージ保持され、生成ロジックより優先される。「review には絶対に触るな」は何世代進化しても守られる。

## 自動ループ: 3 つのトリガー

### 毎夜: ログ監視ジョブ(`bun run evals -- nightly`)

1. **暗黙シグナルの集計**(gear × カテゴリ別):
   - ESC 中断率(`status='client_abort'`)
   - **訂正プロンプト率**: タスク開始から 2 分以内に次の task_event が来て、かつ訂正マーカー(「違う」「そうじゃなくて」「やり直し」等)を含む割合
   - ターン数・トークンの異常(降格ターンの後にリトライが増えていないか)
2. **劣化判定**: 変速が有効なセグメントの訂正率が、変速前の基準線の `correction_rate_jump` 倍を超えたら(n ≥ min_n)、該当ルールを**自動サスペンド**し、changelog 記録 + 通知 + 確認質問をキューに積む
3. **新モデル検知**: requests の `model_requested` に models.yaml の match に該当しないパターンが現れたら(Claude Code のアップデートで新世代モデルが来た合図)、`models.yaml` 更新の TODO 通知 + 次バッチに新 tier のベイクオフ variant を自動追加
4. **ドリフト検知の素材更新**: 日次メトリクスのスナップショット

### 毎週: 評価バッチ + ポリシー再生成

[04](04-evaluation-pipeline.md) の全 stage + stage 7(フィードバック取り込み)。再生成の統治ルール:

- **変更予算**: 1 サイクルのルール変更(追加・削除・閾値変更)は `max_rule_changes_per_batch` 件まで。優先度は「人間フィードバック由来 > 自動ロールバックの復帰判定 > 新規証拠」
- **全変更に根拠を添付**: evidence(win_rate, n, κ)or フィードバック参照 ID
- **changelog + 通知**: 再生成のたびに「今週の変更: docs を Haiku 降格に追加(根拠: 勝率 47%, n=14, 正味トークン -28%)。不満があれば /model-feedback」という 3 行以内のサマリを通知(返答不要のパッシブ通知。これは注意予算外)

### 随時: 自動ロールバック

- 夜間ジョブの劣化判定、または shifter の自動デグレード(エラー率)が発動したら、該当ルールをサスペンドして前バージョンの挙動に戻す
- サスペンドは changelog に origin `auto_rollback` で記録し、**復帰には次バッチでの再検証を要求**する(勝手に再有効化しない)

## モデル世代交代のシナリオ(この仕組みの真価)

次期 Sonnet(仮に claude-sonnet-6)が出た場合の想定フロー:

```
1. Claude Code がアップデートで sonnet-6 を送り始める
   → 夜間ジョブが未知モデルを検知、通知「新モデル検出: claude-sonnet-6」
2. models.yaml の tiers.mid.model を更新(人間、1 行)
   ※ 旧 sonnet-5 は比較用に candidate tier として残す
3. 次の週次バッチが自動で「sonnet-6 baseline vs 旧世代 vs high/low」を再評価
4. 変速ポリシーの前提(昇格・降格の損益分岐)が変わっていれば再生成で自動反映
   例: sonnet-6 が plan でも Opus 並みなら、plan の昇格ルールが自動で消える
5. changelog 通知「sonnet-6 移行に伴い plan の Opus 昇格を停止(根拠: ...)」
6. 違和感があれば /model-feedback で一言 → 次サイクルで反映
```

**人間がやるのは 1 行の設定変更と、気が向いたときの一言だけ。** これが「自己進化」の具体的な姿。

## 実装マッピング

| 機能 | 実装場所 |
|---|---|
| 夜間ジョブ / stage 7(フィードバック取り込み) | `packages/evals`(launchd で夜間起動) |
| 通知 | `scripts/notify.ts`(osascript。review-ui への URL 付き) |
| クイック比較・提案承認・満足度ページ | `packages/review-ui` |
| `/model-feedback` skill | `skills/model-feedback/SKILL.md`(`bun run feedback -- add` を呼ぶ薄いラッパー) |
| 選好キュー・フィードバック・changelog | `packages/datastore`(preference_queue / feedback_notes / policy_changelog) |
| 自動サスペンド | `packages/shifter`(governor)+ 夜間ジョブ |

DB スキーマは [02-data-model.md](02-data-model.md#フィードバックループ関連) 参照。

## この設計の Go/No-Go 指標(M5 ゲート)

- A/B 選好質問の**回答率 ≥ 60%**(それ未満なら質問の選び方 or 頻度が悪い → 予算を減らして質を上げる)
- 満足度チェックで「止めたい」が出ないこと(出たら設計見直し)
- フィードバック → 反映のリードタイム ≤ 1 週間(次バッチで必ず処理される)
- モデル世代交代を 1 回、上記シナリオどおりに乗り切れること
